# SceneWeaver 已知限制

## 当前版本限制

- 前端开发构建可用；release 桌面 EXE 已完成启动验证。`core_smoke` 已覆盖中文/空格路径、增量扫描和 SQLite 持久化；NSIS 安装器已完成专用临时目录的静默安装与启动 smoke。
- FFmpeg / ffprobe 已有 PATH 检测与调用实现，合成 MP4 流程已实测，但尚未作为应用侧车打包；缺失时视频元数据和缩略图会降级。
- 镜头检测当前使用 FFmpeg 场景阈值，并在视频扫描时自动生成代表关键帧、最长 6 秒短预览，以及黑帧比例和相邻像素细节估算的基础质量。代表关键帧下部的亮暗像素分布还会产生保守字幕提示；它不读取文字、不能替代 OCR，真实字幕样式的准确率尚未评估。尚未提供 PySceneDetect 的高级自适应检测、多帧质量筛选或原视频片段播放。FFmpeg 缺失或单个派生步骤失败时保留完整视频片段但没有派生媒体和质量指标。
- 已接入无需下载模型的本地颜色直方图特征，用于已索引图片和用户明确选择的外部参考图之间的“相似”召回；它不能理解中文自然语言、人物或场景语义。
- 可选本地 CLIP ViT-B/32 图文 Provider 已接入：官方 ONNX Runtime Windows x64 CPU DLL 随应用资源分发，模型仍仅在用户点击设置页“下载模型”后联网写入应用数据目录。它不会依赖开发机 Python；模型标记或任一配对缓存文件缺失时会显式降级，不会在扫描或搜索时隐式下载。当前模型的中文语义质量尚未在真实素材集验证，不能视为中文视觉文本模型。
- CLIP 搜索仅对一组明确列出的中文创作/ACG 词（如角色、雨夜、侧脸、回头、战斗、常见景别/视角、动作、情绪和发色）做本地英文提示扩展；Must Not 词不会进入提示。未知中文专有名词与任意句子会保留关键词降级，尚不具备通用中文、日文语义理解。
- 基础自然语言视觉降级仅覆盖红、绿、蓝、黄和夜色等颜色词；对于“人物回头”“城市远景”等通用描述，当前仍只回退到文件名/路径关键词，结果页会如实展示这一限制。
- 视频视觉特征依赖 FFmpeg 成功生成封面；当前开发机已用合成 MP4 验证代表帧索引，真实用户视频素材仍待验证。
- 当前搜索具备本地文件名/路径关键词匹配、颜色词降级召回与参考图相似检索；安装并重建后可使用 CLIP 级图文向量召回。尚不具备经验证的通用中文自然语言视觉语义理解。
- 当前关键词与 CLIP 候选采取去重补充，不是已校准的统一分数融合；CLIP 候选会严格遵守媒体类型、最低质量与 Must Not，但语义层面对 Must 条件、结果多样性和跨路重排仍待完善。
- Must / Must Not 当前过滤文件名、路径、用户确认的 ACG 标签和已生成视频片段的字幕提示；Should 已参与软排序，结果会显示实际命中原因和未满足的 Should，ACG 标签命中不会被误报为文件名命中。关键词、CLIP 与 Entity 候选已在合并后稳定排序；同分候选按素材库轮换以保留基础多样性，但尚未进行跨 Provider 分数校准或镜头级多样性重排。Must Not 的已排除候选原因和 UI 自动过滤仍未完成。
- 可创建自定义选片集合，并支持移动、移除、评分、备注、推荐入/出点和拖拽排序；视频镜头可加入“我的选片”并显示其片段范围。默认和自定义集合均可导出 CSV/JSON/EDL/FCPXML，导出会优先采用推荐入/出点、其次采用片段范围。FCPXML 会规范编码 Windows 文件 URI、使用互不冲突的格式/素材资源 ID，并保留可用的视频源帧率和分辨率；其时间线帧率采用首个有效素材帧率。CMX 3600 EDL 仍固定为 30fps 非丢帧，且两种格式均未处理轨道、转场或速度变化，也尚未在剪辑软件内完成导入验证。
- Entity 支持名称/别名在自然语言请求中的本地匹配及多张正/负本地参考图；正样本可按颜色或（已安装并重建时）CLIP Provider 检索相似素材，负样本以保守权重参与排序惩罚，误加参考图可按实体归属安全移除。结果目前是去重合并而非跨 Provider 校准的统一置信度；实体置信度、通用中文语义效果评估和完整 ACG Creator Pack 仍未完成。
- ACG Creator Pack 的用户确认标签可参与组合搜索，但当前不会自动判断游戏 UI、战斗状态、表情或视角；这些标签必须由用户添加或等待后续视觉模型提供。
- 当前 release EXE 已构建并实际启动验证。NSIS 打包会由 `build.rs` 从锁定的 Cargo WebView2 依赖暂存并显式携带 `WebView2Loader.dll`；当前安装器已完成隔离目录静默安装与 8 秒启动 smoke，并确认 ONNX Runtime 同级分发。尚未在干净 Windows 环境完成交互式安装、卸载、升级和 MSI 验证。

## 开发环境限制

- 使用 MinGW-w64 而非 MSVC，可能存在个别 Windows 原生 API 兼容问题。
- 当前 GNU/Tauri 测试二进制会以 `STATUS_ENTRYPOINT_NOT_FOUND` 启动失败；源码可编译，但 Rust 单元测试尚无法在此环境执行。`core_smoke` 可作为独立二进制实际运行。
- 使用 GNU 工具链时，需要将 `C:\msys64\mingw64\bin` 加入启动 Cargo 的当前 shell `PATH`，以提供 `gcc` 与 `windres`。
- Windows release EXE 与当前代码生成的 NSIS 安装包均已验证；MSI、干净环境的交互式安装、卸载和升级仍待验证。
- 已配置 GitHub Actions Windows 质量门禁。首次远端运行暴露并推动修复了干净工作目录的 `WebView2Loader.dll` 资源缺陷；2026-07-19 的修复复验已通过 Rust 编译和默认 core smoke。它默认不下载语义模型，也不安装 FFmpeg，因此不会替代本地的模型、视频派生或安装器 smoke。
- 镜头检测和关键帧/短预览派生在 90 秒超时后会终止并回收 FFmpeg 子进程；ffprobe 的外部进程超时策略仍待统一到同一监督机制，任务暂停也仍会等待正在执行的单个外部媒体步骤完成或超时。

## 待验证项

- 真实用户视频与更多损坏媒体格式扫描；中文、空格、超过 260 字符路径的图片扫描及单个损坏 PNG 不阻断扫描已由 `core_smoke` 实测通过。
- 外接硬盘离线恢复。
- 任务暂停/恢复/崩溃恢复。
- 无 API Key 核心流程的完整 GUI 端到端验证。
